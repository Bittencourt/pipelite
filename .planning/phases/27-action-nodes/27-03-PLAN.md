  <name>Task 1: JavaScript transform action handler with QuickJS sandbox</name>
  <files>
    src/lib/execution/actions/transform.ts,

  </files>
  <behavior>
    transform.test.ts:
    - Basic transform: code `return { greeting: "Hello " + input.trigger.data.name }` with input.trigger.data.name="World" returns { output: { greeting: "Hello World", logs: [] } }
    - Access previous node output: code `return { total: input.nodes.calc_1.output.value * 2 }` returns correct result
    - console.log capture: code with `console.log("debug"); return { ok: true }` returns { output: { ok: true, logs: ["debug"] } }
    - Multiple console.log calls: all captured in order in logs array
    - MATH helper: code `return { result: MATH.round(3.7) }` returns { output: { result: 4, logs: [] } }
    - TEXT helper: code `return { result: TEXT.upper("hello") }` returns { output: { result: "HELLO", logs: [] } }
    - DATE helper: code `return { today: DATE.today() }` returns a date string
    - LOGIC helper: code `return { result: LOGIC.if(true, "yes", "no") }` returns { output: { result: "yes", logs: [] } }
    - No return value: throws error "Transform must return an object"
    - Returns non-object: throws error "Transform must return an object"
    - Syntax error in code: throws with QuickJS error message
    - Timeout (>5s): code `while(true){}` throws timeout error (use shorter timeout in test)
    - No async: code `return await fetch("x")` fails (no async/network available)
    - No filesystem: code accessing `require` or `process` fails
  </behavior>
  <action>
    1. Create `src/lib/execution/actions/transform.ts`:
       - Import `getQuickJS` from `quickjs-emscripten`, `registerAction` from `./index`
       - Extract FORMULA_FUNCTIONS helper string from formula-engine.ts or duplicate the MATH/TEXT/DATE/LOGIC helper definitions as a const string. These are the same helpers users know from custom field formulas. Include: MATH (abs, ceil, floor, round, min, max, pow, sqrt), TEXT (upper, lower, trim, length, substring, replace, contains, startsWith, endsWith, split, join), DATE (today, now, addDays, addMonths, diffDays, format, parseDate), LOGIC (if, and, or, not, isBlank, isNumber).
       - Handler function:
         a. Extract `code` from config. If empty/missing, throw "No code provided"
         b. Call `getQuickJS()` to get QuickJS instance
         c. Create a new runtime: `QuickJS.newRuntime()`. Set memory limit: `runtime.setMemoryLimit(8 * 1024 * 1024)` (8MB). If setMemoryLimit is not available in v0.32, skip it and rely on interrupt handler only.
         d. Set interrupt handler for 5-second timeout: track start time, return `true` (interrupt) if elapsed > 5000ms. In tests, use a shorter timeout (100ms) for the infinite loop test.
         e. Create context: `runtime.newContext()`
         f. Inject console.log capture: create a `logs` array. Create a JS global `console` object with `log` method that pushes stringified args to the array. Use `vm.newFunction()` for the log callback, `vm.setProp()` to set it on a console object, then `vm.setProp(vm.global, "console", consoleObj)`.
         g. Inject helper functions: eval the FORMULA_FUNCTIONS string in the context
         h. Inject input object: serialize the execution context (trigger + nodes) as JSON, eval `const input = JSON.parse('...')` in the context
         i. Wrap user code in an IIFE: `(function(input) { ${code} })(input)` and eval it
         j. Extract result: if undefined/null, throw "Transform must return an object". If not an object after unwrapping, throw same.
         k. Return `{ output: { ...result, logs } }`
         l. In finally block: `vm.dispose()`, `runtime.dispose()`
       - Register with `registerAction("javascript_transform", handler)`

    2. Update `src/lib/execution/actions/index.ts` to add `import "./transform"` to side-effect imports.

    3. For tests: mock `getQuickJS` only if needed for isolation, but prefer using real QuickJS for integration-level correctness. Set a shorter timeout for the infinite-loop test to avoid slow CI.
  </action>

  <name>Task 2: Webhook response action handler and HTTP handler coordination</name>
  <files>
    src/lib/execution/actions/webhook-response.ts,

  <behavior>
    webhook-response.test.ts:
    - sendWebhookResponse resolves pending waitForWebhookResponse promise with { statusCode, body }
    - waitForWebhookResponse times out after configured ms, rejects with timeout error
    - sendWebhookResponse returns false if no pending response for runId
    - sendWebhookResponse cleans up the pending entry after resolving
    - Webhook response handler: interpolates body values, calls sendWebhookResponse with runId
    - Webhook response handler: returns { output: { sent: true, statusCode, body } }
    - Webhook response handler: if sendWebhookResponse returns false (no pending), still succeeds (output indicates not_waiting)
    - hasWebhookResponseNode correctly detects presence of webhook_response action in workflow nodes
  </behavior>
  <action>
    1. Create `src/lib/execution/actions/webhook-response.ts`:
       - In-memory coordination map: `const pendingResponses = new Map<string, { resolve, reject, timeout }>()`
       - `waitForWebhookResponse(runId: string, timeoutMs = 30_000): Promise<{ statusCode: number; body: unknown }>` -- creates Promise, stores resolve/reject in map with a setTimeout that rejects + cleans up on timeout
       - `sendWebhookResponse(runId: string, statusCode: number, body: unknown): boolean` -- looks up pending, if found: clearTimeout, delete from map, resolve with { statusCode, body }, return true. If not found, return false.
       - `hasWebhookResponseNode(nodes: unknown[]): boolean` -- checks if any node in the array has config.actionType === "webhook_response". Used by webhook route to decide whether to wait.
       - Handler function:
         a. Extract `statusCode` (default 200), `body` from config
         b. Interpolate body values using interpolateDeep with context
         c. Call `sendWebhookResponse(runId, statusCode, interpolatedBody)`
         d. Return `{ output: { sent: result, statusCode, body: interpolatedBody } }`
       - Register with `registerAction("webhook_response", handler)`

    2. Update `src/app/api/webhooks/in/[workflowId]/[secret]/route.ts`:
       - Import `waitForWebhookResponse`, `hasWebhookResponseNode` from `@/lib/execution/actions/webhook-response`
       - Import `executeRun` from `@/lib/execution/engine`
       - After creating the workflow run:
         a. Check if workflow has a webhook_response node: `hasWebhookResponseNode(workflow.nodes as unknown[])`
         b. If YES: execute the run synchronously inline (`await executeRun(run.id)` directly, not via the processor queue). Before executing, register a pending response via `waitForWebhookResponse(run.id, 30_000)`. Use `Promise.race` between waitForWebhookResponse and a 30s timeout. If response received, return `NextResponse.json(response.body, { status: response.statusCode })`. If timeout, return the default `{ ok: true, run_id }` response.
         c. If NO webhook_response node: return immediately with `{ ok: true, run_id }` as before (run executes via processor).
       - The key insight: for webhook-response workflows, we execute synchronously in the HTTP handler so the response can be sent before the connection closes. The execution processor will skip this run since it's already completed.

    3. Update `src/lib/execution/actions/index.ts` to add `import "./webhook-response"` to side-effect imports.
  </action>
