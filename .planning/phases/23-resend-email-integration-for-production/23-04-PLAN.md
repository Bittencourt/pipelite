  <files>src/lib/email-processor.ts, src/lib/email-processor.test.ts, src/app/api/internal/email/process/route.ts, instrumentation.ts</files>

1. Create `src/lib/email-processor.ts`:
   - Follow the exact webhook-processor.ts pattern (setTimeout chaining, no setInterval).
   - `INITIAL_DELAY = 15_000` (15s -- let server finish booting, after webhook processor).
   - `POLL_INTERVAL = 5 * 60_000` (5 minutes).
   - `startEmailProcessor()`: logs "[email-processor] Starting with initial delay of 15s", calls `scheduleTick(INITIAL_DELAY)`.
   - `scheduleTick(delay)`: setTimeout -> fetch POST to `${baseUrl}/api/internal/email/process` with `X-Internal-Secret` header -> schedule next tick.
   - Export `startEmailProcessor`.

2. Create `src/app/api/internal/email/process/route.ts`:
   - Verify `X-Internal-Secret` header matches `process.env.INTERNAL_SECRET`. Return 401 if not.
   - Call `processActivityReminders()` and `processWeeklyDigest()`.
   - Return JSON: `{ reminders: number, digests: number }`.

   **processActivityReminders():**
   - Query activities where: `completedAt IS NULL`, `deletedAt IS NULL`, `reminderSentAt IS NULL`, `dueDate BETWEEN now AND now + 1 hour`.
   - For each activity:
     a. Look up the assigneeId (or ownerId if no assignee) user record for email and locale.
     b. Check notification preferences: skip if `emailActivityReminder` is false.
     c. Set `reminderSentAt = new Date()` on the activity BEFORE sending (prevents duplicates even if send fails).
     d. Call `sendActivityReminderEmail(...)`. Fire-and-forget with `.catch()`.
   - Return count of reminders sent.

   **processWeeklyDigest():**
   - Check if today is Monday. If not, return 0.
   - Check if current hour is between 8-9 UTC (Monday morning window). If not, return 0.
   - **DB-backed deduplication (NOT module-level variable):** Query the `digest_log` table for a row where `weekStart` matches this Monday's date (00:00 UTC). If a row exists, this digest was already sent -- return 0. This survives container restarts per the DB-backed durability pattern established in Phases 19-20.
   - Compute `weekStart` as the previous Monday 00:00 UTC and `weekEnd` as the previous Sunday 23:59 UTC.
   - Query aggregate data for the past week:
     a. New deals: `deals WHERE createdAt >= weekStart AND createdAt < weekEnd`
     b. Deals that changed stage: count deals updated this week (exact stage-change tracking would require audit log which doesn't exist).
     c. Won deals: `deals WHERE status = 'won' AND updatedAt >= weekStart`
     d. Lost deals: `deals WHERE status = 'lost' AND updatedAt >= weekStart`
   - Query per-user activity data:
     a. Overdue activities: `activities WHERE dueDate < now AND completedAt IS NULL AND deletedAt IS NULL AND (assigneeId = userId OR ownerId = userId)`
     b. Upcoming this week: `activities WHERE dueDate BETWEEN now AND weekEnd AND completedAt IS NULL AND deletedAt IS NULL AND (assigneeId = userId OR ownerId = userId)`
   - Get all users with `emailWeeklyDigest = true` (or no preference row, since default is true): LEFT JOIN users with notificationPreferences, filter where `emailWeeklyDigest IS NULL OR emailWeeklyDigest = true`.
   - For each opted-in user: call `sendWeeklyDigestEmail(...)` with the aggregate deal data (same for all users) plus per-user activity counts. Fire-and-forget.
   - **After sending all digests:** Insert a row into `digest_log` with `sentAt = new Date()`, `userCount = count`, `weekStart = thisMonday`.
   - Return count of digests sent.

3. Update `instrumentation.ts`:
   - Add email processor startup after webhook processor:
     ```typescript

     startEmailProcessor()
     ```

4. Update `src/lib/email-processor.test.ts`: replace placeholder tests with actual tests for the pure logic parts:

   - Test that non-Monday days return 0 for digest processing
   Extract `isMondayMorning(date: Date): boolean` and `getWeekBoundaries(date: Date): { weekStart: Date, weekEnd: Date }` as pure helper functions for testability.

Note: Deal summary data (new, won, lost) is global (same for all users). Activity data (overdue, upcoming) is per-user. Run deal queries ONCE, then loop users for activity queries. This follows the pitfall guidance from RESEARCH.md about avoiding N queries for deal data.
