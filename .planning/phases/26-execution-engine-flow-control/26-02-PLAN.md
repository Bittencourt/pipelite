  <files>src/lib/execution/engine.ts, src/lib/execution/engine.test.ts</files>
  <behavior>
    - Linear 3-node graph executes all nodes and completes run
    - Condition node with true match executes true branch, skips false branch
    - Condition node with false match executes false branch, skips true branch
    - Delay node with future time sets run to waiting and persists context
    - Delay node with past time (null from resolver) skips delay and continues
    - Node execution error fails the run with descriptive message
    - Resume from waiting: engine picks up from currentNodeId after delay
  </behavior>
  <action>
    1. Write tests FIRST in `src/lib/execution/engine.test.ts` with integration-style tests covering all behaviors above. Tests should mock the DB layer (or use a test helper that stubs Drizzle calls) to keep them fast. Focus on verifying the engine logic rather than full DB integration. Run tests -- must fail (RED).

    2. Create `src/lib/execution/engine.ts` with the core execution logic:

    3. `executeRun(runId: string)` -- main entry point:
       - Load the workflow run and its workflow from DB
       - Build a nodeMap (Map<string, WorkflowNode>) from workflow.nodes JSONB
       - Load or initialize ExecutionContext from run.context column. If resuming (run has currentNodeId), start from the node AFTER the delay node (the delay node's nextNodeId). If fresh run, start from nodes[0].
       - Walk nodes in a while loop via `currentNodeId` pointer

    4. For each node in the walk:
       - Create a workflow_run_step record with status "running", input as the current context snapshot
       - Dispatch by node.type:
         - "action": For now, action nodes are a stub/passthrough -- set output to `{ type: node.config.actionType ?? "unknown", status: "stub" }`. Phase 27 implements real actions. Complete the step.
         - "condition": Call `evaluateCondition(node.config, context)`. Execute the matching branch (trueBranch or falseBranch) by calling a helper `executeBranch(branchStartNodeId, nodeMap, context, runId)` which walks nodes linearly until nextNodeId is null. After branch completes, continue from the condition node's `nextNodeId` (merge point).
         - "delay": Call `resolveDelay(node.config, context)`. If result is null (past time), skip the delay -- complete the step and continue. If result is a Date, set the step to "waiting" with resume_at, set the run to "waiting" with currentNodeId = node.nextNodeId (where to resume after delay), persist context to run.context, and RETURN (exit the execution loop -- processor will resume later).
       - After dispatch, add node output to context.nodes[node.id], complete the step, persist context to run.context, advance to node.nextNodeId

    5. Error handling: wrap each node execution in try/catch. On error, fail the step (status "failed", error message), fail the run (status "failed", error message including node id and label), and return.

    6. When the while loop ends (currentNodeId is null), set run status to "completed" with completedAt.

    7. Helper `executeBranch(startNodeId: string | null, nodeMap, context, runId)`:
       - If startNodeId is null, return (empty branch)
       - Walk nodes linearly from startNodeId, creating steps, executing them, until nextNodeId is null
       - Branch nodes are all action/delay type (no nested conditions in v1 -- keep it simple)
       - If a delay is hit inside a branch, persist and return (same as main loop)

    Use Drizzle for all DB operations. Import from `@/db` and `@/db/schema/workflows`.

    8. Run tests -- all must pass (GREEN).
  </action>

  <behavior>
    - Claims a pending run when no run for that workflow is currently running or waiting (happy path)
    - Does NOT claim a pending run when another run for the same workflow is already running (serial enforcement)
    - Does NOT claim a pending run when another run for the same workflow is in waiting status (serial enforcement includes waiting)
    - Drains queue: claims multiple pending runs from different workflows in a single tick
    - Resumes a waiting run whose resume_at timestamp has elapsed
    - Does NOT resume a waiting run whose resume_at is still in the future
    - Calls executeRun with the correct run ID after claiming or resuming
  </behavior>

       - Verify the atomic claim SQL is called with the correct serial enforcement subquery
       - Verify processWaitingRuns queries for steps past resume_at and transitions them
       - Verify executeRun is called with correct run IDs
       - Verify the drain loop stops when no more runs are claimed
       Run tests -- must fail (RED).

    2. Create `src/lib/execution/execution-processor.ts` mirroring the schedule-processor pattern:
       - Constants: INITIAL_DELAY = 5_000 (5s), POLL_INTERVAL = 5_000 (5s -- faster than schedule processor since execution should be responsive)
       - `startExecutionProcessor()`: logs startup, calls scheduleTick(INITIAL_DELAY)
       - `scheduleTick(delay)`: setTimeout with async callback that calls processPendingRuns() then processWaitingRuns(), catches errors, always schedules next tick
       - `processPendingRuns()`:
         - Atomic claim via raw SQL: `UPDATE workflow_runs SET status = 'running', started_at = NOW() WHERE id = (SELECT wr.id FROM workflow_runs wr WHERE wr.status = 'pending' AND NOT EXISTS (SELECT 1 FROM workflow_runs wr2 WHERE wr2.workflow_id = wr.workflow_id AND wr2.status IN ('running', 'waiting')) ORDER BY wr.created_at ASC LIMIT 1 FOR UPDATE SKIP LOCKED) RETURNING *`
         - If a run is claimed, call `executeRun(run.id)` from engine.ts
         - Loop: keep claiming until no more pending runs are available (drain the queue per tick)
       - `processWaitingRuns()`:
         - Query workflow_run_steps where status = 'waiting' AND resume_at <= NOW()
         - For each: update step status to "running", update the parent run status to "running", then call `executeRun(run.id)` (engine reads currentNodeId to know where to resume)
         - Use `FOR UPDATE SKIP LOCKED` on the step query to prevent duplicate resume
