  <behavior>
    - Fixed mode: 30 minutes -> resumeAt is ~30 min from now
    - Fixed mode: 2 hours -> resumeAt is ~2 hours from now
    - Fixed mode: 3 days -> resumeAt is ~3 days from now
    - Until mode: future ISO datetime -> resumeAt is that datetime
    - Until mode: past ISO datetime -> resumeAt is null (skip delay, execute immediately)
    - Field mode: valid timestamp field in context -> resumeAt is that timestamp
    - Field mode: field not found -> throws error with clear message
    - Field mode: field is not a valid date -> throws error with clear message
    - Any mode: computed resumeAt > 30 days from now -> capped at 30 days
    - Until mode: invalid datetime string -> throws error
  </behavior>
  <action>
    1. Write tests FIRST in `delay-resolver.test.ts`. Use `vi.useFakeTimers()` to control Date.now() for deterministic assertions. Test all three modes, the 30-day cap, and error cases. Run tests -- must fail (RED).

       - `resolveDelay(config: DelayConfig, context: ExecutionContext): Date | null`
       - Fixed mode: use date-fns `addMinutes`/`addHours`/`addDays` based on unit
       - Until mode: parse `config.untilTime` as Date, return null if in the past
       - Field mode: use `resolveFieldPath` from condition-evaluator to get value, validate it's a parseable date, return as Date
       - For all modes: if computed date is > 30 days from now, cap at exactly 30 days from now
       - Throw descriptive errors for invalid inputs (not silently fail)
